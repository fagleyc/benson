# microWakeWord — "Hey Benson" training

This directory holds the data, scripts, and (eventually) the trained model
for the household-specific wake word "Hey Benson". It lives next to
`/opt/benson/scripts/ears/respeaker-kitchen.yaml`, which is the device-side
YAML.

## Why is this here?

The ReSpeaker Lite runs **on-device microWakeWord** (mWW). The wake-word
model is a tiny TFLite spectrogram classifier baked into the ESPHome
firmware. Today the kitchen unit listens for `Okay Nabu` (Home Assistant's
default brand). We want `Hey Benson`.

There are two paths to get there:

1. **Pretrained model** (~30 minutes of work, no training). Use
   `hey_jarvis`, already included in formatBCE's base YAML. Closest
   household-assistant "feel" of the pretrained set. No training needed.
2. **Custom model** (~hours of work, may need Colab + real voice data).
   Train a `hey_benson.tflite` and host it for the ESPHome integration to
   download.

Right now this directory is set up for **path 2**, but **path 1 is the
recommended starting point** — flash with `hey_jarvis`, get the pipeline
working end-to-end, then come back here to upgrade to `hey_benson` once
you have real family voice samples.

```
  work/

    microWakeWord/                     # cloned, installed -e into venv
    basic_training_notebook.ipynb      # reference: the Colab the team ships
    generated_augmented_features/      # mmap'd positive spectrograms
    trained_models/hey_benson/         # checkpoints + final .tflite
    train.log                          # most recent training run log
    training_parameters.yaml           # generated by train_hey_benson.py
  synthetic_positive/
    hey_benson/         hey_benson_extra1/  hey_benson_extra2/  hey_benson_extra3/
    all/                                 # 3,500 merged piper-TTS clips
  data/
    mit_rirs/                            # 270 room impulse responses
    negative_datasets/                   # pre-built mmap negative spectrograms
      speech/  no_speech/  dinner_party/  dinner_party_eval/
  models/                                # final hey_benson.tflite + .json land here
  train_hey_benson.py                    # local training driver
  README.md                              # this file
```

Total disk: ~25 GB (the unzipped negative spectrogram features are
14 GB by themselves; this overshoots the 5 GB budget in Casey's brief
because the mmap spectrograms cannot be re-zipped — they are designed
to be memory-mapped). Free space is 3.4 TB so it's fine in practice.

## Current status (2026-05-17)

A first-pass `hey_benson.tflite` (62 KB) was trained locally on the Spark
from 3,500 synthetic libritts_r positives + the HF negative spectrogram
sets, ~4,000 training steps. Streaming-quantized ROC sweep results:

```
Cutoff 0.93:  frr=0.0000;  faph=0.000
Cutoff 0.63:  frr=0.0000;  faph=0.187
Cutoff 0.49:  frr=0.0000;  faph=0.562
```

(Full sweep in `models/hey_benson_roc.txt`.) On synthetic test positives
+ the dinner_party_eval ambient set the model is well inside Casey's
quality target (frr < 30 %, faph < 1.0/hr). **But these positives are
TTS-generated by the same engine that produced training data, so the
recall number is optimistic.** Real-world FRR on actual family voices
is the unknown — expect somewhere between 5 % and 40 %, hence the
manifest is shipped at `probability_cutoff: 0.97` (conservative) so
false-accepts stay near zero while we collect real-world recall data.

`models/hey_benson.tflite` and `models/hey_benson.json` are ready to
host + reference from `respeaker-kitchen.yaml`. **Strong recommendation:**
flash with `hey_jarvis` first (already in formatBCE's base), verify the
end-to-end pipeline, *then* swap in `hey_benson` and tune the cutoff
based on real family experience over a few weeks.

## Quick path (RECOMMENDED for tonight): flash with `hey_jarvis`

The formatBCE base YAML already enables `hey_jarvis`. In Home Assistant,
go to the Voice Assistant pipeline and pick `hey_jarvis` as the wake
word for this satellite. Done. No training, no flash changes.

If you want to silence the other built-ins (okay_nabu, hey_mycroft, kenobi):
disable them in the HA wake-word selector for the pipeline — easier than
forking the base YAML.

## Slow path: train your own `hey_benson`

### 1. Best results: kahrendt Colab + real family voices

The mWW README is explicit that synthetic-only training "most likely
[produces] a model [that] will not be usable." Real recordings from each
family member dramatically improve recall. Plan:

1. Get each family member to record ~50 utterances of "Hey Benson" with
   their phone's voice-memo app (vary pitch, distance, room, mood).
   Trim each clip to ~1.5 sec. Save as WAV @ 16 kHz mono. Aim for **at
   least 200 total real clips** across the household.
2. Open the Colab:
   <https://colab.research.google.com/github/OHF-Voice/micro-wake-word/blob/main/notebooks/basic_training_notebook.ipynb>
   (raw notebook source:
   <https://github.com/OHF-Voice/micro-wake-word/blob/main/notebooks/basic_training_notebook.ipynb>)
3. In the Colab, change the runtime to a **T4 GPU** (free tier is fine,
   training on CPU is unreasonably slow).
4. Replace the piper-generated positive-samples step with an upload of
   your real family clips into the `generated_samples/` directory. Keep
   piper-generated samples too — mix both pools (real samples weighted
   higher).
5. Run all cells through training. Adjust `negative_class_weight` higher
   (try 30–40) if false-accepts are a problem after testing.
6. Download `stream_state_internal_quant.tflite` at the end.

### 2. Local synthetic-only attempt (what's set up here)

```bash
source /opt/benson/microwakeword/venv/bin/activate
python /opt/benson/microwakeword/train_hey_benson.py
```

This produces `work/trained_models/hey_benson/tflite_stream_state_internal_quant/
stream_state_internal_quant.tflite`. Quality will be marginal — this
is meant as a sanity check, not a final model. **TensorFlow on aarch64
ships CPU-only on this box, so the training run takes several hours.**

Inspect the printed test metrics. If the model is good enough to ship
(false-reject < 30%, false-accept < 1/hour against the dinner_party
test set), copy it to `models/hey_benson.tflite` and create the manifest
JSON below.

## Manifest JSON template

Drop next to the `.tflite`:

```json
{
  "type": "micro",
  "wake_word": "Hey Benson",
  "author": "Casey Fagley",
  "website": "https://github.com/caseyfagley/benson",
  "model": "hey_benson.tflite",
  "trained_languages": ["en"],
  "version": 2,
  "micro": {
    "probability_cutoff": 0.95,
    "feature_step_size": 10,
    "sliding_window_size": 5,
    "tensor_arena_size": 30000,
    "minimum_esphome_version": "2024.7.0"
  }
}
```

Notes on the fields:

- `probability_cutoff` — raise to reduce false-accepts, lower if the
  model misses real triggers. 0.9–0.97 is the typical band.
- `tensor_arena_size` — 30000 matches kenobi; if ESPHome refuses to load
  the model, bump up by 5000 at a time until it accepts.
- `feature_step_size` and `sliding_window_size` — match the
  `window_step_ms` (10) and `sliding_window_size` (5) used during
  training. Do not change unless you re-train with different values.

## Hosting + flashing

The ESPHome integration downloads models by URL at flash time. Options:

- Push `models/hey_benson.tflite` + `hey_benson.json` to a GitHub repo
  and use the `https://raw.githubusercontent.com/.../hey_benson.json`
  URL in `respeaker-kitchen.yaml` (see the wake-word comment block at
  the bottom of that file for the exact diff).
- Or self-host on the Spark — `nginx` or a simple `python -m http.server`
  on a LAN address works, but ESPHome compiles models in at build time,
  not at runtime, so any reachable HTTPS URL is fine.

## Failsafe: a bare "Benson" wake word

Casey mentioned wanting bare "Benson" too. Strongly recommend **against**
that as a standalone model. "Benson" is a surname (Benson Boone, etc.),
a casual interjection in some dialects, and a single trochaic word — too
short for mWW to discriminate against close-by negatives. Train one model
for `Hey Benson`, set its probability_cutoff aggressively, and revisit
bare "Benson" only after months of real-world tuning data.
